## 1. Context（背景与目标）

### 1.1 现状

本仓库是 pnpm + Turborepo monorepo，三层架构：

- **前端** `apps/web`：React 19.2 + Vite 8 + Tailwind 4 + Tiptap 3 + react-router-dom 7，通过 SSE（fetch + ReadableStream）消费流式响应。
- **后端** `apps/api-server`：NestJS 11 + Prisma 7 + Redis，作为前端与 Python Worker 之间的消息中转（请求队列 `agent:request:list` → 响应队列 `agent:response:list:<sessionId>`）。
- **Python Agent** `agents/`：LangChain 1.3 + LangGraph 1.2 + Qdrant + Redis，支持 exec / plan / multi 三种模式。
- **共享包** `packages/shared`：跨端类型。

### 2.2 API Key 加密存储（AES-256-GCM）

**新增工具**（`apps/api-server/src/common/crypto.ts`）：
- 使用 Node.js 原生 `crypto`（不引入 crypto-js，减少依赖）。
- `encrypt(plain: string): string`：AES-256-GCM，返回 `iv:tag:ciphertext`（均 base64）。
- `decrypt(payload: string): string`：解析并解密。
- 密钥从 `process.env.ENCRYPTION_KEY`（32 字节 hex）读取，启动时校验存在性。

**改造 ModelConfigService**：
- `create` / `update`：`apiKey` 非空时 `encrypt()` 后存库。
- 新增 `getDecryptedApiKey(config)`：内部方法，读取时 `decrypt()`。
- `getAll` / `getSelected` / `getById`：**响应中剔除 `apiKey` 明文**（用 `select` 排除，或映射时删除字段），前端只显示 `hasApiKey: boolean`。
- `ChatController.stream`：取配置后调用 `getDecryptedApiKey` 获取真实 key 传给 Worker。

**环境变量**：`.env.example` 新增 `ENCRYPTION_KEY=`（注释说明用 `openssl rand -hex 32` 生成）。

### 2.4 ChatService 并发瓶颈修复

**问题**：`this.reader` 是单例，多个并发请求的 `blPop` 争抢同一连接，导致串行化。

**修复**（`chat.service.ts`）：
- 移除类成员 `reader`，仅保留 `publisher`（单例足够，lPush 非阻塞）。
- `processMessage` 内部创建独立 reader：`const reader = createClient({ url: redisUrl }); await reader.connect();`，在 `finally` 中 `reader.quit()`。

- `onModuleDestroy` 仅断开 `publisher`。

**注意事项**：每次请求新建连接有开销，但 BLPOP 必须独占连接，这是 Redis 队列模式的固有代价。P2 可考虑改用 Redis Streams + consumer group 优化。

## 4. P2 阶段：性能优化（后续迭代，大纲）

- **React 渲染优化**：`MessageBubble` `React.memo` + `useCallback` 稳定回调；`MessageList` 虚拟化（`@tanstack/react-virtual`）应对长会话。
- **SSE 流式批量 flush**：前端 `processChunk` 用 `requestAnimationFrame` 或 16ms 节流，合并多个 token 后一次性 setState，避免每个 token 浅拷贝整个 messages 数组。
- **模型配置缓存**：`ModelConfigService` 内存缓存（Map + TTL），`selectOne` 时失效；前端 `useModelConfigs` 减少重复请求。
- **Prisma 连接池**：`PrismaPg` 配置 `max` 连接数；`$transaction` 超时设置。
- **SSE 背压**：`ChatController.stream` 监听 `res.write` 返回值（false 表示内核缓冲满），暂停消费 generator，`drain` 事件恢复。
- **Redis Streams**：请求/响应队列从 List+BLPOP 升级为 Streams+XREADGROUP，支持 consumer group、ACK、重放，消除每请求新建连接的开销。
- **记忆持久化**：`MemoryManager._turns` 从内存 dict 迁移到 Redis hash，Worker 重启不丢失。
