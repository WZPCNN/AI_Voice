**Why**: API Key 需加密存储；当前无异常统一处理，错误堆栈泄露给客户端。

**Files & Changes**:
1. **新建** `apps/api-server/src/common/crypto.ts`
   - 使用 Node.js 原生 `crypto`（不引入 crypto-js，减少依赖）
   - `encrypt(plain: string): string`：AES-256-GCM，返回 `iv:tag:ciphertext`（均 base64）
   - `decrypt(payload: string): string`：解析并解密
   - 密钥从 `process.env.ENCRYPTION_KEY`（32 字节 hex）读取，模块加载时校验长度
   - 导出 `maskApiKey(key: string): string`：脱敏显示（前4后4，中间 ****）

   - 实现 `ExceptionFilter`，捕获所有异常
   - 生产环境隐藏堆栈，仅返回 `{ code, message, timestamp }`
   - 开发环境附带 `stack`
   - Prisma 已知错误码映射：P2002(唯一约束)→409，P2025(未找到)→404
   - HttpException 透传其 status & response

3. **修改** `apps/api-server/src/main.ts`
   - `app.useGlobalFilters(new AllExceptionsFilter())`
   - 端口改 `const port = app.get(ConfigService).get('PORT') ?? 4000`

## 五、假设与决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 状态管理 | zustand | 已安装、兼容 React 19、用户确认 |
| 认证 | JWT + bcrypt | 用户指定 |
| 加密 | AES-256-GCM（Node 原生 crypto） | 无需额外依赖 |
| Python 版本 | >=3.12,<3.15（CI 用 3.14） | torch 兼容性 |
| 执行范围 | P0 + P1 + CI/CD 一次执行 | 用户确认 |
| P2/P3 | 本次不执行 | 用户选 P0+P1+CI/CD；P2 性能优化融入 P1-3 组件拆分 |
| ChatService 并发 | 每请求独立 Redis reader 连接 | 避免单例争抢 |
| API 前缀 | 全局 /api | 统一规范 |
| 提示词 | 模块化 prompts/ 包 | 替代单文件常量 |

**风险提示**：
- schema.prisma 添加 enum 需要 `prisma migrate dev`，需数据库可用
- 拆分 ChatPage 时需保持 SSE 流式逻辑正确性（最复杂步骤）
- Python 44 处 print 替换需逐一核对上下文
